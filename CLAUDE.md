# CLAUDE.md

Этот файл содержит инструкции для Claude Code (claude.ai/code) при работе с этим репозиторием.

## Запуск приложения

```bash
# Активировать venv (Windows / bash)
source .venv/Scripts/activate

# Запустить Flask-сервер на 0.0.0.0:5000
python main.py
```

Файла `requirements.txt` нет. Зависимости установлены в `.venv/`: Flask 3.1, Flask-Login, Flask-WTF, SQLAlchemy 2.0, sqlalchemy-serializer, Werkzeug, requests. Если добавляешь новую зависимость — поставь её в `.venv` и подумай о том, чтобы создать `requirements.txt`.

Тестов, линтера и шага сборки в проекте нет.

## Архитектура

Это небольшое Flask-приложение, работающее как **мост между личным Android-устройством (с MacroDroid) и веб-панелью** для просмотра уведомлений из мессенджеров. Интерфейс и тексты — на русском.

### Привязка устройства (главная нетривиальная идея проекта)

1. Пользователь регистрируется через `/register` → сервер генерирует 8-значный `connect_code` и сохраняет его в строке `User`.
2. Пользователю показывается код на странице `/code` (автообновление каждые 3 секунды).
3. Android-устройство (макрос MacroDroid) делает `GET /connect?code=<код>`. Сервер запоминает `request.remote_addr` как `tablet_ip` пользователя.
4. Как только `tablet_ip` заполнен, `/code` редиректит на `/home`.
5. Затем устройство шлёт уведомления из мессенджеров через `POST /add` (поля формы: `sender`, `text`, `messenger_name`). Сервер определяет пользователя **по совпадению `request.remote_addr` с `User.tablet_ip`** — никакого токена авторизации на `/add` нет. Это означает, что IP устройства в локальной сети должен оставаться стабильным, иначе сообщения будут привязаны не к тому пользователю (или потеряются).
6. Сохранённые сообщения отображаются на `/messages` (автообновление каждые 5 секунд через meta refresh).

### Слой данных

- **SQLite** в файле `db/blogs.db`. Путь захардкожен в [main.py:17](main.py#L17).
- Engine создаётся с `check_same_thread=False`, потому что в [main.py:18](main.py#L18) при импорте создаётся одна глобальная сессия `db_sess`, переиспользуемая всеми запросами. Имей в виду: конкурентные записи или долгие запросы через эту общую сессию могут дать трудноуловимые баги. В новых эндпоинтах лучше создавать сессию на каждый запрос через `db_sessions.create_session()` (так уже делает неиспользуемый блюпринт `data/user_api.py`).
- Модели лежат в [data/users.py](data/users.py) (`User`, `Messages`). Новые модели нужно импортировать в [data/__all_models.py](data/__all_models.py), иначе `SqlAlchemyBase.metadata.create_all` не создаст для них таблицы при `global_init`.
- `data/db_sessions.py` предоставляет `global_init(db_file)` и `create_session()`. `global_init` идемпотентна — повторный вызов сразу возвращает управление, если фабрика уже создана.

### Маршруты

Все маршруты живут в [main.py](main.py). Авторизация — через сессию (`flask.session['user_id']`); декоратора `@login_required` нет, каждый защищённый маршрут проверяет `session.get('user_id')` вручную.

- `/`, `/home`, `/login`, `/register`, `/logout`, `/code` — страницы.
- `/connect` (GET) — эндпоинт привязки устройства, дёргается из MacroDroid.
- `/add` (POST) — приём сообщений, дёргается из MacroDroid; пользователь определяется по `remote_addr`.
- `/messages` — панель просмотра.

### Неиспользуемый код

- В [data/user_api.py](data/user_api.py) объявлен REST-блюпринт, но он **не зарегистрирован** в `main.py` и обращается к полям (`about`), которых нет в модели `User`. Считай это мёртвым кодом, пока явно не работаешь над API.
- В [forms/user.py](forms/user.py) есть `RegisterForm` (Flask-WTF), но `/register` парсит `request.form` напрямую и форму не использует. В `main.py` импортируется, но не задействуется.
- В [data/users.py](data/users.py) закомментированы наброски модели `Chats` и связанных полей.
